# 架构与迁移范围

## 部署模型

首个生产部署包含三个可独立部署的 Go 二进制程序：

- **API** 负责同步 HTTP 请求：身份认证、运营后台、大厅、配置、游戏指令和钱包指令。
- **Worker** 负责可靠的异步任务：轮次状态流转、结算、佣金、排行榜、outbox 发布、充值确认和提现对账。
- **实时网关** 负责 WebSocket 连接、房间、在线状态和客户端通知。

它们从同一源代码仓库和同一个 PostgreSQL 集群起步。每个模块都有明确的职责边界，并通过应用接口或可靠事件进行通信。只有当模块需要独立扩缩容、部署、持久化或故障隔离时，才将其拆分为独立服务。

## 当前系统能力归属

| 现有能力 | 新归属模块 | 主要一致性规则 |
| --- | --- | --- |
| 账号、游客、密码及第三方登录 | Identity | 会话必须可撤销，密码必须使用 Argon2id，严禁使用 MD5。 |
| 玩家与 GM API | API / Operations | 对特权指令实施 RBAC，并保留不可变审计日志。 |
| 余额、扣除投注、派奖、退款、充值及提现 | Wallet | 账本记录与余额更新必须处于同一个 PostgreSQL 事务中。 |
| 轮次、停止投注、TRON/K 线结果及结算 | Game / Worker | 通过轮次版本和唯一结算键保证结算幂等。 |
| 代理层级与返水 | Agent / Worker | 佣金来源必须是已结算投注，且只能被唯一记录一次。 |
| 聊天、客服、红包及推送 | Realtime | 先持久化消息记录，再通过 Redis/NATS 扇出。 |
| TRON 监听、K 线订阅和 PQPA 回调 | Chain / Worker | 必须验证回调签名、服务商事件 ID 和交易哈希，并保证幂等。 |
| 上传与对象存储 | Operations | 存储凭证必须仅保留在服务端，上传使用短时有效的签名 URL。 |
| 定时任务与脚本 | Worker | 使用可靠的任务状态、重试、退避和死信处理机制。 |

## 必需的事务模式

处理投注请求时，API 必须在**同一个**事务中锁定钱包行、校验可用余额、创建投注、追加账本扣款记录、更新钱包余额，并追加一条 outbox 事件。Worker 只能处理已提交的 outbox 记录。这样可以避免将数据库写入和 Redis 发布误认为同一操作而产生的数据不一致问题。

## 计划中的接口

- 外部 API：采用 REST/JSON 和 OpenAPI，使用 JWT 访问令牌与幂等键。
- 内部同步调用：模块尚未独立部署时优先使用 Go 接口；独立部署后使用 ConnectRPC/gRPC。
- 内部异步调用：使用 NATS JetStream 主题，例如 `game.bet.placed`、`game.round.settled`、`wallet.ledger.committed`、`chain.deposit.credited`。
- 实时协议：使用经认证的 WebSocket 消息，并明确版本化事件名称。

## 拆分触发条件

1. 当连接数量需要独立扩缩容时，拆分 `realtime`。
2. 当服务商故障或签名控制需要隔离时，拆分 `chain`。
3. 当轮次处理需要独立容量或发布节奏时，拆分 `game-settlement`。
4. 只有在账本接口稳定且能独占数据库所有权后，才拆分 `wallet`。
